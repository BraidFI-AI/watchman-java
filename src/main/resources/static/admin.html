üìä Phase Scoring Mechanics ‚ñº

Purpose

Technical reference for understanding the scoring lifecycle. Each phase represents a step in the sequential process that entity data takes from raw input to final match decision. Some phases contribute numerical scores, others prepare or filter data.

Scoring Formula Overview

Final Score = Weighted Average of Active Factors

finalScore = (nameScore √ó nameWeight + addressScore √ó addressWeight + ...) / totalWeight

Weights are only included when a factor is present and non-zero. This prevents dilution by missing data.

Default Weights:

  • Name: 35
  • Address: 25
  • Critical ID (gov/crypto/contact): 50
  • Supporting Info (dates): 15

What is a Phase?

A phase represents a step in the scoring lifecycle - the sequential journey that entity data takes from raw input to final match decision.

Total: 12 phases (defined in Phase.java enum)
Traced: 10 phases (write debug entries when trace=true)
Not Traced: 3 phases (execute but don't write trace entries)

Phase Hierarchy

Top-Level Phases (10 traced):

  • NORMALIZATION - Text cleanup
  • NAME_COMPARISON - Primary name matching
  • ALT_NAME_COMPARISON - Alternate name matching
  • GOV_ID_COMPARISON - Government ID matching
  • CRYPTO_COMPARISON - Cryptocurrency address matching
  • CONTACT_COMPARISON - Email/phone matching
  • ADDRESS_COMPARISON - Geographic matching
  • DATE_COMPARISON - Birth date matching
  • AGGREGATION - Weighted score combination
  • FILTERING - Applied in SearchController post-scoring

Child Processes (2 not traced):

  • TOKENIZATION - Generates word combinations (runs inside NAME_COMPARISON/ALT_NAME_COMPARISON)
  • PHONETIC_FILTER - Soundex-based filtering (runs inside NAME_COMPARISON/ALT_NAME_COMPARISON)

Score Contributors

8 phases directly contribute numerical scores:

  • NAME_COMPARISON, ALT_NAME_COMPARISON (contribute to nameScore)
  • GOV_ID_COMPARISON, CRYPTO_COMPARISON, CONTACT_COMPARISON (contribute to criticalIdScore)
  • ADDRESS_COMPARISON (contributes to addressScore)
  • DATE_COMPARISON (contributes to supportingInfoScore)
  • AGGREGATION (combines weighted scores)

All 12 phases execute during scoring. Tracing affects observability, not functionality.

Key Phases Detail

Phase 1: NORMALIZATION

Purpose: Text cleanup to enable accurate comparison

  • Removes diacritics (√© ‚Üí e, √± ‚Üí n)
  • Converts to lowercase
  • Removes punctuation
  • Normalizes whitespace

Impact on score: NONE (preprocessing only)

Phase 2: TOKENIZATION

Purpose: Handle name particles and spacing variations

Generates word combinations by merging short words (‚â§3 chars):

  • "Jean de la Cruz" ‚Üí ["jean de la cruz", "jean dela cruz", "jean delacruz"]
  • "van der Berg" ‚Üí ["van der berg", "vander berg", "vanderberg"]

Impact on score: INCREASES matches by 20-40%

Phase 3: PHONETIC_FILTER

Purpose: Skip obviously non-matching comparisons early

Compares first character Soundex codes. If first characters are NOT phonetically similar, returns score 0.0.

Impact on score: REDUCES false positives by ~5%

Can be disabled: config.phoneticFilteringDisabled = true

Phase 4: NAME_COMPARISON

Purpose: Primary identity matching using Jaro-Winkler similarity

Impact on score: DOMINANT factor (weight 35). Typically 60-80% of final score.

Score range:

  • 1.0 = Exact match
  • 0.85-0.99 = Very close (common typos, accents)
  • 0.70-0.84 = Similar but noticeable differences
  • <0.70 = Probably different entities
‚öôÔ∏è ScoreConfig Reference ‚ñº

Summary

Centralized type-safe configuration for scoring parameters. Configuration is managed through Spring Boot's @ConfigurationProperties system.

SimilarityConfig - Algorithm Parameters

10 parameters controlling Jaro-Winkler scoring behavior:

jaroWinklerBoostThreshold (default: 0.7)

Minimum score before Jaro-Winkler prefix boost applies. Range: 0-1.

jaroWinklerPrefixSize (default: 4)

Number of leading characters to compare for prefix boost.

lengthDifferencePenaltyWeight (default: 0.3)

Penalty weight when names differ significantly in length. Range: 0-1.

lengthDifferenceCutoffFactor (default: 0.9)

Threshold ratio for applying length penalty. Range: 0-1.

differentLetterPenaltyWeight (default: 0.9)

Penalty when characters differ between names. Range: 0-1.

exactMatchFavoritism (default: 0.0)

Boost applied to exact matches. Range: 0-1.

unmatchedIndexTokenWeight (default: 0.15)

Weight for tokens in query not found in index. Range: 0-1.

phoneticFilteringDisabled (default: false)

Disable Soundex phonetic matching. May reduce fuzzy match quality.

keepStopwords (default: false)

Keep common words like 'the', 'and' in matching (usually filtered).

logStopwordDebugging (default: false)

Enable debug logging for stopword filtering behavior.

WeightConfig - Business-Level Controls

13 parameters for scoring weights and phase toggles:

Weights (4 parameters)

  • nameWeight (default: 0.4) - Weight for name comparison scores
  • addressWeight (default: 0.3) - Weight for address comparison scores
  • criticalIdWeight (default: 0.2) - Weight for government ID matching
  • supportingInfoWeight (default: 0.1) - Weight for additional data (crypto, contact, dates)

Thresholds (2 parameters)

  • minimumScore (default: 0.7) - Minimum score to return match
  • exactMatchThreshold (default: 0.95) - Score considered exact match

Phase Toggles (7 parameters)

Enable/disable comparison phases (all default: true):

  • nameComparisonEnabled - Primary name matching
  • altNameComparisonEnabled - Alternate name matching
  • addressComparisonEnabled - Geographic matching
  • govIdComparisonEnabled - Government ID matching
  • cryptoComparisonEnabled - Cryptocurrency address matching
  • contactComparisonEnabled - Email/phone matching
  • dateComparisonEnabled - Date of birth matching

Configuration Sources (Priority Order)

  1. Command-line: --watchman.similarity.jaro-winkler-boost-threshold=0.8
  2. Environment: WATCHMAN_SIMILARITY_JARO_WINKLER_BOOST_THRESHOLD=0.8
  3. YAML: application.yml
  4. Defaults: SimilarityConfig.java / WeightConfig.java

Total configuration surface: 23 parameters (10 SimilarityConfig + 13 WeightConfig)

üîç ScoreTrace Guide ‚ñº

Summary

Opt-in scoring observability system capturing phase-by-phase scoring decisions. Generates JSON traces (technical) and HTML reports (compliance/audit). Zero overhead when disabled.

10 Traced Phases

These phases write debug entries when trace=true:

  1. NORMALIZATION - Text cleanup and preparation
  2. NAME_COMPARISON - Jaro-Winkler on primary name
  3. ALT_NAME_COMPARISON - Match against alternate names
  4. GOV_ID_COMPARISON - TIN, passport, national ID
  5. CRYPTO_COMPARISON - Cryptocurrency addresses
  6. CONTACT_COMPARISON - Email, phone
  7. ADDRESS_COMPARISON - Geographic matching
  8. DATE_COMPARISON - Date of birth with transposition detection
  9. AGGREGATION - Weighted score combination
  10. FILTERING - Applied in SearchController post-scoring

3 Untraced Phases

These execute but don't write trace entries:

  • TOKENIZATION - Runs inside NAME_COMPARISON/ALT_NAME_COMPARISON (child process)
  • PHONETIC_FILTER - Runs inside NAME_COMPARISON/ALT_NAME_COMPARISON (child process)
  • FILTERING - Runs in SearchController after scoring (post-processing)

Note: The 3 untraced phases execute and function correctly. Tracing affects observability (debug output), not functionality.

API Integration

Search with trace:

curl "http://localhost:8080/v1/search?name=Maduro&trace=true"

Response includes reportUrl field

Get HTML report:

curl "http://localhost:8080/api/reports/abc-123"

Get JSON summary:

curl "http://localhost:8080/api/reports/abc-123/summary"

Implementation Details

  • Trace storage: In-memory with 24-hour TTL
  • Performance: <5ns overhead when disabled
  • Key classes: ScoringContext.java, EntityScorer.java, TraceSummaryService.java

Usage Pattern

// Enable tracing
ScoringContext ctx = ScoringContext.enabled(sessionId);
double score = entityScorer.scoreWithBreakdown(query, candidate, ctx);
ScoringTrace trace = ctx.toTrace();

// Disabled (production default)
ScoringContext ctx = ScoringContext.disabled();
double score = entityScorer.scoreWithBreakdown(query, candidate, ctx);
// No trace overhead
